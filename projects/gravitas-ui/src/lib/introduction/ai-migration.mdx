  ## Recommended migration order

  <div className="gv-doc-callout-list">
    {[
      [
        'Start with buttons',
        'Replace `btn` markup and local action wrappers with `gv-button` and `gv-icon-button` first.',
      ],
      [
        'Then move forms',
        'Migrate text inputs, textarea, select, checkbox, radio, and switch controls while preserving Angular form bindings.',
      ],
      [
        'Then shared chrome',
        'Replace alerts, badges, cards, loaders, empty states, avatars, and dividers where the application is duplicating those patterns.',
      ],
      [
        'Then structured widgets',
        'Move tabs, tables, modals, dropdowns, and tooltips only when the screen fits the supported Gravitas contract.',

  ## Default replacement map

  <div className="gv-doc-grid gv-doc-grid--wide">
    <div className="gv-doc-card">
      <h3 className="gv-doc-card-title">Straight replacements</h3>
      <ul className="gv-doc-list">
        <li>`button.btn*` → `gv-button`</li>
        <li>icon-only action buttons → `gv-icon-button`</li>
        <li>`input.form-control` → `gv-input`</li>
        <li>`textarea.form-control` → `gv-textarea`</li>
        <li>`select.form-select` and simple select wrappers → `gv-select`</li>
        <li>`input[type="checkbox"]` → `gv-checkbox`</li>
        <li>`input[type="radio"]` → `gv-radio`</li>
        <li>binary settings toggles → `gv-switch`</li>
      </ul>
    </div>

    <div className="gv-doc-card">
      <h3 className="gv-doc-card-title">Common UI patterns</h3>
      <ul className="gv-doc-list">
        <li>alert banners → `gv-alert`</li>
        <li>badges and counters → `gv-badge`</li>
        <li>content containers → `gv-card`</li>
        <li>simple dialogs → `gv-modal`</li>
        <li>tab systems → `gv-tabs`, `gv-tab-list`, `gv-tab`, `gv-tab-panels`, `gv-tab-panel`</li>
        <li>repeat loading placeholders → `gv-loader` and `gv-skeleton`</li>
        <li>empty results blocks → `gv-empty`</li>
      </ul>
    </div>
  </div>

  ## Form migration rules

  <div className="gv-doc-panel">
    <ul className="gv-doc-list">
      <li>Preserve `formControlName`, `[formControl]`, or `[(ngModel)]` bindings.</li>
      <li>Preserve validators, disabled state, touched state, and async form behavior.</li>
      <li>Move helper and error messaging into Gravitas component inputs where supported.</li>
      <li>Do not wrap self-rendering Gravitas field controls in `gv-form-field` unless there is a real gap to solve.</li>
      <li>Do not leave duplicated helper or error markup in place after migration.</li>
    </ul>
  </div>

  ## Import strategy

  <div className="gv-doc-panel">
    <p className="gv-doc-section-copy" style={{ marginTop: 0 }}>
      The agent should import only the standalone components needed by each feature component.
    </p>

```ts
import { Button, InputComponent, Select, Tabs, TabList, Tab, TabPanels, TabPanel } from 'gravitas-ui';
```

    <p className="gv-doc-section-copy" style={{ marginBottom: 0 }}>
      Do not blanket-import the entire library into every screen. Keep imports explicit so usage
      stays readable and tree-shakable.
    </p>
  </div>

  ## What the agent must not guess

  <div className="gv-doc-grid gv-doc-grid--wide">
    <div className="gv-doc-card">
      <h3 className="gv-doc-card-title">Do not auto-replace</h3>
      <ul className="gv-doc-list">
        <li>third-party data grids</li>
        <li>date pickers without a Gravitas equivalent</li>
        <li>autocomplete and async typeahead widgets</li>
        <li>rich text editors</li>
        <li>deeply custom tables or tree grids</li>
        <li>multi-step workflow modals with embedded app logic</li>
      </ul>
    </div>

    <div className="gv-doc-card">
      <h3 className="gv-doc-card-title">Report instead</h3>
      <ul className="gv-doc-list">
        <li>which controls were migrated</li>
        <li>which Gravitas components were introduced</li>
        <li>which legacy wrappers or styles were removed</li>
        <li>which widgets were left in place intentionally</li>
        <li>which unsupported patterns need follow-up work</li>
      </ul>
    </div>
  </div>

  ## Bootstrap guidance

  <p className="gv-doc-section-copy">
    The target state is not “remove Bootstrap from the application.” Gravitas is intentionally
    Bootstrap-aligned. The agent should keep Bootstrap for layout and shell structure, but stop
    using raw Bootstrap component primitives like `btn`, `form-control`, `form-select`, and
    `input-group` when Gravitas already owns that UI pattern.
  </p>
