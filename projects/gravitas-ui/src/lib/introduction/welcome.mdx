## Why Gravitas exists

<p className="gv-doc-section-copy">
  When teams build UI without a shared system, the same problems show up quickly: slightly different
  buttons, inconsistent spacing, repeated form patterns, duplicated styling logic, and components
  that behave differently depending on who built them.
</p>

<p className="gv-doc-section-copy">
  Gravitas exists to reduce that drift. The goal is to provide a set of shared foundations and
  reusable components so interfaces feel related, predictable, and easier to maintain over time.
</p>

## Expectations when contributing

<p className="gv-doc-section-copy">
  A component is not finished just because it renders correctly once.
</p>

<p className="gv-doc-section-copy">
  When adding or updating something in Gravitas, its Storybook entry should be updated too. That
  includes realistic stories, key states, and documentation that reflects how the component is
  actually meant to be used.
</p>

<p className="gv-doc-section-copy">
  Good entries usually show the primary use case, supported variants, important states such as
  disabled, loading, empty, or error, and any accessibility notes that help another developer use
  the component correctly.
</p>

## Themes and density

<p className="gv-doc-section-copy">
  Use the toolbar at the top of Storybook to switch theme and density. These global controls are
  part of the review process, not just decoration. Components should be checked in more than one
  presentation mode, especially when working on forms, tables, and dense application layouts.
</p>
